# 通用代码规范

> **作用**：为科研项目中的代码修改提供最小、可执行的质量基线。它不替代项目现有规范，也不要求把所有代码改成同一种风格。

---

## 使用方式

本文档会随项目初始化生成，但为了保持工作代理的上下文最小化，不会自动加入每个任务。

- 涉及代码、脚本或配置修改，且没有更具体的项目规范时，主代理应把本文档加入实现或检查上下文；
- 已有后端、前端、包级或团队规范时，优先加入更具体的规范，按需再引用本文档；
- 当前任务还涉及特定语言时，从 `../languages/`（语言书写规范目录）只选择对应的一份规范；
- 文献检索、纯运行、纯报告等不修改代码的工作不需要加载本文档。

这使它成为代码任务的默认候选基线，而不是所有代理的隐式长上下文。

---

## 规则优先级

发生冲突时，按以下顺序处理：

1. 项目现有的 `AGENTS.md`、`CONTRIBUTING.md`、格式化配置、检查配置和真实代码模式；
2. 当前任务的范围、研究方案、项目约束和完成条件；
3. 当前语言或框架已经选定的公开规范与工具配置；
4. 本文档的通用基线。

不要以“符合通用规范”为理由覆盖项目已有约定，也不要进行与任务无关的大面积格式化或重构。

## 修改前

- 先读取当前任务的目标、范围、完成条件和相关规范；
- 搜索已有实现，至少查看一个相近示例；
- 明确输入、输出、错误路径和验证方式；
- 科研改动涉及数据集、数据切分、模型结构、损失函数、指标、基线或主张范围时，先按任务流程记录计划变更；
- 不确定需求、方案或研究假设时，先讨论并记录决定，不用代码猜测答案。

## 实现规则

- 名称表达业务或研究含义，避免无语义缩写和隐式全局状态；
- 一个模块或函数只承担清晰、可说明的职责；出现重复模式时优先复用现有实现，不为一次性代码过早抽象；
- 在外部边界处理类型、格式和错误：文件、网络响应、命令输出、配置和未类型化数据都不能直接假定正确；
- 配置、常量和路径集中管理；不要散落重复值、机器绝对路径或环境专属开关；
- 错误信息说明失败的操作和可处理原因，不泄露密钥、私有数据、完整输入或内部路径；
- 只修改当前任务允许的文件。新增依赖、配置和接口时，同时说明用途、兼容性影响和验证方式。

## 科研代码与可复现性

- 把实验输入、参数、随机种子、环境版本、运行命令和产物位置记录在可追踪文件中；
- 数据读取、预处理、训练、评估和报告应保留可复跑入口，避免依赖人工临时步骤；
- 指标应明确计算对象、数据切分、聚合方式和比较基线；
- 结果文件应能关联回对应代码版本、配置和运行记录；
- 单次运行只是运行结果，不自动等同于证据充分、任务关闭或科研主张成立。

## 测试与验证

- 测试可观察行为和失败边界，不只测试实现细节；
- 修复缺陷时，优先增加能复现该缺陷的回归测试；
- 运行项目已有的格式化、静态检查、类型检查、构建和目标测试；没有现成命令时，只补与当前改动直接相关的最小验证；
- 在任务记录中保存实际运行的验证命令、结果和必要产物引用；
- 审查时检查范围漂移、错误处理、测试覆盖、可复现性和研究结论边界。

## 安全与协作

- 密码、访问令牌、私有数据和 `.env` 文件只通过安全的运行环境提供，绝不写入源码、测试、日志、任务记录或提交；
- 外部输入按不可信数据处理；涉及删除、覆盖、网络上传、发布和付费资源时先确认边界；
- 遵从项目的提交、版本、许可证和贡献规范；没有项目约定时，提交应保持小而聚焦，并说明行为变化与验证结果；
- 代码注释解释不明显的设计原因、研究限制或反直觉行为，不重复代码字面意思。

## 完成前检查

- [ ] 修改符合当前项目的真实代码模式和任务范围；
- [ ] 没有秘密信息、绝对个人路径或未声明的环境依赖；
- [ ] 已运行适用的格式化、检查、类型检查、构建或测试；
- [ ] 科研代码的配置、种子、输入和产物可以追溯；
- [ ] 验证结果和不确定项已写入任务记录；
- [ ] 没有把单次运行或聊天声明当成任务完成或科研结论。

## 公开基线

本文档参考以下公开资料的共同原则；具体语言规范应由项目明确选择后再写入语言或模块规范：

- `EditorConfig`（跨编辑器格式规则）：https://editorconfig.org/
- `OWASP Secure Coding Practices`（安全编码实践）：https://owasp.org/www-project-secure-coding-practices-quick-reference-guide/
- `Best Practices for Scientific Computing`（科研计算最佳实践）：https://doi.org/10.1371/journal.pbio.1001745
- `Good enough practices in scientific computing`（科研计算最低充分实践）：https://doi.org/10.1371/journal.pcbi.1005510

对 TypeScript、Python、Go、Rust、Java、C#、C++、Kotlin 或 Swift 项目，应在对应规范中记录采用的官方风格指南、格式化工具和静态检查工具，而不是在这里混合多套互相冲突的细则。
