# Hermes 五角色迁移说明

## 变化

正式角色收敛为 `planner`（规划代理）、`researcher`（检索代理）、`coder`（编码代理）、`runner`（运行代理）和 `reviewer`（复核代理）。

`Role`（角色）决定权限、工具、输出契约和独立性要求；`Profile`（模式）决定本次工作的关注点。模式不会生成新的角色文档或平台代理模板。

正式五角色模板只提供给 `Claude Code`（Claude 代码工具）和 `Codex`（代码代理平台）。其他平台继续保留原版通用代理和兼容行为。

## 旧名称

| 旧名称 | 规范化结果 |
| --- | --- |
| `scientist`（科学家） | `planner:research_design`（规划代理的研究设计模式） |
| `builder`（构建代理） | `coder:implementation`（编码代理的实现模式） |
| `literature`（文献代理） | `researcher:literature`（检索代理的文献模式） |
| `research/scout`（检索侦察代理） | 根据目标选择检索代理模式，默认 `codebase`（代码库） |
| `evaluator`（评估代理） | `reviewer:evidence`（复核代理的证据模式） |
| `claim-reviewer`（主张审查代理） | `reviewer:claim`（复核代理的主张模式） |
| `analyst`（分析代理） | 根据目标选择规划或复核模式，无法判断时使用 `planner:root_cause`（规划代理的根因模式） |

旧任务卡读取时会给出弃用提示，但不会重写追加式历史。新任务卡写入时只保存正式角色和合法模式。

`evidence-curator`（证据整理代理）不再允许新派发。产物路径、哈希、运行清单和引用检查由 `evidence.py`（证据工具）确定性完成，证据质量判断由 `reviewer:evidence`（证据复核模式）完成。

## 派发上下文

从 `v0.7.1-beta.0`（首个测试版）开始，主代理先生成 validated dispatch（已验证派发），其中只保留：

1. `job_id`（工作编号）和 `hermes_revision`（Hermes 修订号）；
2. 当前角色与模式；
3. 当前工作包或任务级空绑定；
4. 有界目标；
5. 最多 3 个直接引用。

完整需求、设计、历史报告、事件、账本和全部规范只在当前检查确实需要时读取。

## 测量

| 项目 | 改造前 | 改造后 | 变化 |
| --- | ---: | ---: | ---: |
| 角色文档 | 8 个 / 9673 字符 | 6 个 / 6819 字符 | 字符减少 29.5% |
| `Claude Code`（Claude 代码工具）正式 Hermes 模板 | 7 个 / 9873 字符 | 5 个 / 3865 字符 | 字符减少 60.9% |
| 两平台正式 Hermes 模板合计 | 7 个 / 9873 字符 | 10 个 / 8904 字符 | 新增 Codex 支持后仍减少 9.8% |
| 单个正式模板平均长度 | 1410 字符 | 890 字符 | 减少 36.9% |

派发正文测试限制在 2000 字符内。`Claude Code`（Claude 代码工具）只向 `Agent`（代理）传 `job_id`（工作编号），`Codex`（代码代理平台）使用同一份紧凑派发和结果契约，直接在当前项目工作区协作。

## 兼容边界

- 原版任务、原版通用代理和非 closure 任务行为不变。
- 已有 closure 任务缺少修订号时按 0 读取，下一次语义写入会补齐并递增。
- 旧角色模板不再随新安装分发；已安装的旧别名只在 Hermes closure 任务中映射到正式角色和模式。
- 证据、产物、主张、批准和任务事件的现有语义不变。
