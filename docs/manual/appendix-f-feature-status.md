# 附录 F 功能状态、安全与许可证

## 目标

明确当前版本中已实现、实验性和未实现的边界，并说明安全、许可证和上游署名要求。

## 适用范围

适用于 `0.7.1-beta.0`（测试版），后续版本应重新核对。

## 前置条件

以当前仓库代码、帮助、模板和测试为准，不以上游规划代替事实。

### 安全边界

- `allowed_commands`（允许命令）和代理文件范围用于流程约束，不是操作系统级沙箱。
- `RecordBus`（记录总线）使用本地 `JSONL`（逐行 JSON）文件，便于审计，但不能防止有文件权限的进程篡改历史。
- 平台钩子可以未启用、被环境变量关闭或因平台差异失效。执行危险命令前仍需人工确认。
- 缺少新鲜钩子时三种 closure 模式都会提示降级；`standard`（标准）和 `publication`（发表）仍通过更严格的运行、证据、统计、主张和人工审批记录完成审计。
- 不要把 `.env`（环境变量文件）、访问令牌、密码、私有数据或未脱敏实验输入写入任务、记录和报告，也不要提交到 `Git`（版本控制系统）。
- 外部文献、数据、模型和网络服务需要单独核对许可证、隐私和访问权限。

### 许可证与上游署名

- 本项目基于 `mindfold-ai/Trellis`（上游项目）修改，不是上游官方版本，也不代表上游立场。
- 项目和两个发布包的 `SPDX`（许可证标识）以及根说明统一使用 `AGPL-3.0-or-later`（AGPL 第 3 版或任何后续版本），与上游 `COPYRIGHT`（版权声明）中“version 3 or any later version”（第 3 版或任何后续版本）的授权口径一致。
- 根目录 `LICENSE`（许可证原文）和 `COPYRIGHT`（版权声明）必须随分发内容保留；发布预检只核对元数据与说明的一致性，不改写或缩减上游权利文本。
- 发布修改版时应明确标注修改事实和日期，保留上游版权信息，并提供许可证要求的对应源码获取方式。
- `research-trellis`（CLI 包）和 `research-trellis-core`（核心包）的打包清单都必须包含 `LICENSE`（许可证）和 `COPYRIGHT`（版权声明）；发布预检会对缺失文件失败关闭，不改写上游权利文本。

## 预期结果

使用者不会把存在命令误解为生产级安全能力，也不会把未实现功能写入操作流程；分发前能够找到上游来源、许可证原文和待确认的口径差异。

## 失败恢复

发现手册与新版本不一致时，以新版本帮助、实现和测试为准，并更新本页状态。
