# 01 项目概览

## 目标

理解 `Research Trellis`（科研改造版）解决什么问题、与上游的关系，以及当前版本不能提供什么。

## 适用范围

适合希望把人工智能协作过程写入仓库、保留任务规划、实验运行和证据记录的科研开发项目。第一次试用应选择可用 `Git`（版本控制工具）回滚的低风险项目。

## 前置条件

- 能管理目标项目中的文件和提交。
- 接受当前版本为 `0.6.0-beta.31`（测试版）。
- 不把它当作安全隔离工具或自动科研结论系统。

## 操作步骤

1. 把它理解为叠加在现有项目和人工智能工具上的本地工作流，而不是新的编辑器或网页界面。
2. 区分两套约定：普通任务使用 `planning`（规划中）、`in_progress`（进行中）等实际状态；`planning -> running -> review -> claim_ready -> approved`（规划、运行、复核、待审批、已审批）是科研流程协议模板，当前没有通用运行时强制每一次迁移。
3. 按以下边界决定是否采用：

| 状态 | 当前含义 |
| --- | --- |
| 已实现 | 命令、模板或脚本存在，并有测试或本次运行验证 |
| 实验性 | 已能运行，但整个分支仍为测试版，真实项目中需要人工复核 |
| 未实现 | 当前代码没有提供，不能从上游规划或名称推断为可用 |

当前核心能力是把以下内容保存到项目中：

- `.trellis/spec/`（项目规范目录）保存真实项目约定；
- `.trellis/tasks/`（任务目录）保存需求、设计、实施计划和上下文清单；
- 每个任务下的 `hermes/`（科研记录目录）保存实验配置、运行清单、证据、主张、比较和报告；
- 平台目录保存技能、命令、代理和钩子模板。

## 预期结果

你能判断该工具是否适合当前项目，并明确知道：它提高的是记录和检查的可追踪性，不会自动保证科研结论正确。

## 失败恢复

若项目不可回滚、包含未隔离的秘密信息，或不能接受测试版行为，先不要初始化。已经初始化但决定退出时，先阅读[更新、恢复与卸载](16-update-recovery-and-uninstall.md)，不要直接运行卸载命令。
